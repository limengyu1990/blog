Plutus笔记
发布于：2021-04-14 22:24:48 标签：/ Plutus / DApps / 智能合约 / Cardano / 访问：

Plutus平台为Cardano区块链提供本地智能合约语言。要理解并精通Plutus，必须理解三个概念：

  • 扩展的UTXO模型（EUTXO）

  • Plutus Core - Plutus的”链上”部分

  • Plutus应用程序框架（PAF）‒ Plutus的”链下”部分，支持与智能合约的交互。

Plutus合约由在区块链上运行的部分（链上代码）和在用户机器上运行的部分（链下代码或客户端代码）组成。链上和链下的代码都是用Haskell编写的，而Plutus智能合约实际上是有效的Haskell程序。链下代码可以使用PAF编写，然后由GHC（General Haskell编译器）编译，而链上代码（使用Plutus Core编写）则由Plutus编译器编译。

理解这些Plutus概念和本机令牌功能之间的关系至关重要，以了解它们的交互如何将后者变成更有用和更强大的功能。

The Extended UTXO model

Cardano（像彼特比一样）使用未使用的（U）事务（TX）输出（O）记帐模型。在UTXO模型中，事务有输入和输出，其中输入是以前事务的未使用输出。一旦一个输出被用作事务中的输入，它就被消耗掉了，再也不能使用了。输出由一个地址（公钥或公钥散列）和一个值（由一定ada和可选的附加本机令牌量组成）指定。输出的地址决定允许哪些事务“解锁”输出并将其用作输入。事务必须由地址对应的私钥的所有者签名。
把地址想象成一把”锁”，只有用正确的”钥匙”即正确的签名才能”解锁”。

EUTXO模型在两个方向上扩展了该模型:

  • 它用锁和钥匙的类比来概括地址的概念。EUTXO模型中的地址可以包含脚本形式的任意逻辑，而不是限制对公钥的锁和对签名的键。例如，当节点验证事务时，该节点确定是否允许该事务使用某个输出作为输入。事务将查找由输出地址提供的脚本，如果事务可以使用输出作为输入，则将执行脚本。

  • UTXO和EUTXO的第二个区别是，除了地址和值之外，输出还可以携带（几乎）任意数据。通过允许脚本携带状态，这使得脚本更加强大。

在验证地址时，脚本将访问输出所携带的数据、验证的事务以及事务为每个输入提供的一些称为redeemers的附加数据。通过查找所有这些信息，脚本有足够的上下文在可能非常复杂的情况和用例中给出”是”或”否”的答案。总之，EUTXO扩展了UTXO模型，允许输出地址包含复杂的逻辑来决定哪些事务可以解锁它们，并向所有输出添加自定义数据。

与其他会计模型相比，EUTXO模型具有独特的优势。 交易验证的成功或失败仅取决于交易本身及其输入，而不取决于区块链上的其他任何内容。 因此，可以在将交易发送到区块链之前在链外检查交易的有效性。 如果某个其他事务同时消耗了该事务预期的输入，则该事务仍可能失败，但是如果所有输入仍然存在，则该事务可以保证成功。

这与基于帐户的模型（以太坊使用的模型）形成对比，在该模型中，事务可能在脚本中期执行中失败。 这在EUTXO中永远不会发生。 此外，交易执行成本可以在传输前在链外确定-这是以太坊上无法实现的另一功能。

最后，由于事务验证的“本地”性质，因此高度并行是可能的：原则上，如果一个节点不尝试消耗相同的输入，则节点可以并行验证事务。 这对于效率和推理都非常有用，它简化了对可能结果的分析，并证明了”没有什么坏事”会发生。 您可以在上一篇博客文章中更深入地研究EUTXO模型。

Plutus Core

为了实现EUTXO模型，有必要清楚地定义script和data这两个术语。脚本需要一种明确的、指定良好的脚本语言，定义附加到输出并用作redeemers的数据类型也很重要。

这就是Plutus Core的用武之地。Plutus Core是Cardano使用的脚本语言。它是一种简单的函数式语言，类似于Haskell，Haskell的一个很大的子集可以用来编写Plutus Core脚本。作为智能合约作者，你不会写任何Plutus Core。所有Plutus Core程序都是由Haskell编译器插件生成的。

这些脚本将在链上的事务验证“活动”期间由节点执行。它们将以验证程序脚本的形式或作为铸币策略锁定UTXO，以控制本机令牌的铸币和刻录（请参阅下文）。

Redeemer数据是一种简单的（代数）数据类型，可以在Haskell中轻松定义，这也是为什么Haskell是编写Plutus Core脚本的好选项的另一个原因。 实际上，智能合约开发人员将在Haskell中编写validator(验证程序)脚本，然后将其自动编译到Plutus Core中。

适当的Haskell库通过提供验证期间检查事务的核心数据类型，并通过提供许多助手函数和更高级别的抽象，使合约作者能够专注于业务逻辑，而不必担心太多的低级细节，从而简化了编写此类验证逻辑。

The Plutus Application Framework (PAF)

验证程序脚本的链上状态只能由花费和产生脚本输出的事务修改。在编写Plutus应用程序时，我们不仅需要考虑应用程序的链上部分（Plutus Core脚本），还需要考虑构建和提交事务的链下部分。

链下代码是用Haskell编写的，就像链上代码一样。这样我们只需要编写一次业务逻辑。然后我们就可以在validator脚本和构建运行validator脚本的事务的代码中使用它。

许多应用程序需要监视UTXO集以查看特定地址的更改，因此，如果我们将合同写为状态机，则需要跟踪代表状态机当前状态的未花费输出，并当链上状态更新时更新我们本地的状态。同样，许多应用程序需要与电子钱包后端进行通信，以访问其用于交易的加密货币。

PAF提供了对Plutus应用程序常用的服务的轻松访问。使用框架库部署的应用程序可以在Plutus应用程序后端上运行，后者为访问区块链和其他关注点（如持久性、日志记录和监控）提供运行时支持。在PAF之上编写的应用程序自动提供HTTP和WebSocket接口，可用于从web浏览器与应用程序交互。

Native tokens

任何用户都可以创建自己的令牌，并且可以像ada一样自由地发送和接收令牌。每个本机令牌都有自己的铸造策略，该策略决定了可以minted(铸造)和burnt(烧制)令牌的条件。

目前，这些铸币策略由指定签名和时间锁的简单规则组合而成。例如，策略可以声明仅允许由五个可能的签名中的两个签名的事务创建或刻录令牌。另一个策略只允许在特定时间段之前或之后造币。

尽管这些基本的构建块功能强大，但它们并没有涵盖所有可能的用途。例如，使用这样简单的策略对不可替代令牌（nft）进行建模是可能的，但很尴尬。这可以通过使用timelock来创建NFT来完成，方法是将minting操作限制在特定的时间点。如果在到达该时间点之前只铸造了一个令牌，则该令牌在技术上是不可替代的（因为只有一个）。但要检验这一点，仅仅检验铸币政策是不够的。我们需要查看代币的铸造历史，以确保它确实只铸造过一次。

随着Plutus的部署，用户将能够使用Plutus核心编写铸币策略。在minting或burning期间，Plutus Core策略脚本将在minting或burning事务的上下文中执行，脚本必须批准或禁止该操作。这将进一步加速卡达诺”NFTs”的发展，因为这将促成更为复杂的铸币政策的制定，并允许以不可信任的方式创建”NFTs”。

Other

Transaction output

事务输出包括一个地址（可以认为是锁）和一个值。与此类似，属于地址的签名是解锁输出的密钥。一旦解锁，输出就可以用作输入。新事务使用以前事务的输出，并生成可由将来事务使用的新输出。每个UTXO只能消费一次，并且作为一个整体。每个输出只能由一个输入使用，而且只能由一个输入使用。

Transaction input

事务输入是前一个事务的输出。事务输入包括一个指针和一个用作解锁密钥的加密签名。指针指向上一个事务输出，键解锁这个输出。当输出被输入解锁时，区块链将解锁的输出标记为“spent已用”。给定事务创建的新输出可以被新的输入指向，因此链条继续。这些新的输出（尚未解锁，即已使用）是UTXOs。Unspent(未使用的)输出只是那些尚未spent(使用的)输出

在UTXO会计模型中，事务消耗以前事务的unspent(未用)输出，并生成新的输出，这些输出可以用作将来事务的输入。

validation scripts

验证脚本的物理位置是每个节点：在验证新事务时，节点将运行验证程序，并且仅在验证脚本成功时才接受事务。

wallet contracts

“钱包合同”的物理位置是”用户钱包”。一些进程运行在用户的机器上（或者可能在用户的webbrowser中，尽管我认为还没有实现）。本地进程当然可以有一个web接口。

回滚交易的唯一方法是创建区块链的分支-经过几次”确认”之后，这并不容易，实际上是不可能的。

Datum/Redeemer/Context

Datum只是包含在UTxO内部的一条数据，因此可用于将状态保持在智能合约中。
可用于向Validator脚本提供更多信息(例如，也可用于针对Redeemer进行验证)

Redeemer data是由想要花费一些硬币的人提供的。在验证器逻辑内部，我们可以使用此数据执行一些检查。例如，如果某人想要访问合同中查看的资金，则他可能需要提供密码或其他内容（这取决于您的用例）。
Redeemer是一种逻辑，用于验证UTXO是否可以被spent/burned等
用于验证调用脚本的用户是否可以解锁资金。(例如，用于解锁的密码)

Context只是被验证的事务，由validator节点提供。在它里面，您可以访问交易的输入和输出以及进行交易的人。
包含有关交易的信息.

好吧，交易已经由消费者产生(他的钱包为他产生了交易)，然后他提交了交易，验证器”捡起”这个交易就是context（验证器脚本的第三个参数）。
plutus脚本包含您作为智能合约开发人员编写的任意逻辑，您可以自由使用datum、redember和context(事务)，以任何方式允许或拒绝提供给您的plutus脚本的事务。

说明

正如我之前所解释的，为了使某些事情发生在区块链上，必须进行交易。更改状态的唯一方法是向区块链添加新交易。
例如，您可以做的是编写一个稍微复杂些的验证程序，该验证程序允许提供者始终取回资金。 或者可能是在达到特定的插槽编号后检索它们。 您无法自动执行“链上”操作，但是可以通过对钱包进行编程来自动执行“链下”操作，该方法将等待截止日期，然后自动提交检索交易。

您必须小心“合同”的含义。 如果您指的是链上验证程序（Plutus脚本），则否定-验证程序是一种被动的事情，无法“发送资金”。 如果您的意思是代码在钱包中脱链运行，那么可以：您可以编写一个将休眠一段时间的钱包合同，然后检查资金是否已被索取，如果没有，请提交检索交易。

发送到节点的每个事务都包含一个“SlotRange”，此范围表示如果当前插槽不属于此范围，则事务无效。另一方面，“mustValidateIn”是在事务发送之前设置其有效范围的方法，正如Lars所解释的，默认的“SlotRange”是“always”。在授予合同的情况下，我们用mustValidateIn将SlotRange设置为from now，这样就可以与截止日期进行比较。

它是消费交易的一部分，但也可以选择是生产交易的一部分。因此，如果您想使用一个位于脚本地址的UTxO，您可以在事务中包含散列到该地址的脚本。
